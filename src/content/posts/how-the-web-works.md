## TL;DR

웹 서비스는 **프론트엔드(브라우저) - API - 백엔드(서버) - 데이터베이스** 흐름으로 동작한다. 프론트엔드 개발자라도 이 전체 흐름을 이해해야 효율적인 디버깅, API 설계 협업, 성능 최적화가 가능하다. NEWNDY 프로젝트의 Google 로그인 구현을 예시로, 웹의 동작 원리를 단계별로 정리한다.

## 웹의 기본 구조: 클라이언트-서버 모델

웹의 모든 상호작용은 **요청(Request)**과 **응답(Response)**의 반복이다.

```
브라우저 (프론트엔드)  ───  인터넷  ───  서버 (백엔드)
   사용자가 보는 화면               데이터를 처리하는 곳
```

집으로 비유하면 이해가 쉽다:

| 웹 개념 | 비유 |
|---|---|
| **HTML** | 집의 뼈대 — 벽, 문, 창문 위치 |
| **CSS** | 인테리어 — 색깔, 크기, 간격 |
| **JavaScript** | 집의 기능 — 문 열고 닫기, 전등 켜기 |
| **백엔드** | 수도관, 전기, 가스 — 보이지 않지만 동작시키는 것 |
| **데이터베이스** | 창고 — 데이터를 보관하는 곳 |

## API — 프론트엔드와 백엔드를 잇는 통로

API(Application Programming Interface)는 **두 시스템 간의 약속된 통신 규칙**이다.

유튜브로 예를 들면:

```
1. 브라우저가 유튜브 서버에 요청
   "id가 abc123인 영상 데이터 줘"

2. 유튜브 서버가 응답
   "제목: OOO, 조회수: 100만, 영상파일: ..."

3. 브라우저가 받은 데이터로 화면을 그림
```

이 **1번과 2번 사이의 통로가 바로 API**다. 프론트엔드 개발자가 API를 이해해야 하는 이유는 명확하다:

- **에러 디버깅**: 네트워크 탭에서 응답 상태 코드를 읽고 문제를 추적
- **백엔드 협업**: API 명세를 기반으로 요청/응답 구조를 합의
- **성능 최적화**: 불필요한 API 호출을 줄여 로딩 속도 개선

## 실제 사례: NEWNDY Google 로그인 플로우

NEWNDY 프로젝트에서 구현한 Google 로그인의 전체 플로우를 통해, 웹의 동작 구조를 구체적으로 살펴보자.

```
1. "Google로 로그인" 버튼 클릭            ← [FE] 사용자 액션
        ↓
2. Google 로그인 페이지로 이동             ← [FE → Google] 리다이렉트
        ↓
3. Google이 "이 사람 맞아" 확인            ← [Google] 인증 처리
        ↓
4. NestJS 서버로 "확인됐어" + 사용자 정보 전달  ← [Google → BE] 콜백
        ↓
5. 로그인 완료 → 토큰 발급 → 클라이언트 전달  ← [BE → FE] 응답
```

이 과정에서 프론트엔드는 **1번(UI), 2번(리다이렉트), 5번(토큰 저장 및 상태 관리)**를 담당하고, 백엔드(NestJS)는 **3~4번의 인증 로직**을 처리한다.

## 웹 개발의 전체 기술 스택 위치

```
┌─────────────────────────────────────────────────┐
│  프론트엔드 (Next.js + React + TypeScript)      │
│  → 사용자가 보는 화면, 상호작용, 상태 관리      │
├─────────────────────────────────────────────────┤
│  API 계층 (REST API / HTTP)                     │
│  → 프론트엔드와 백엔드 사이의 통신 규칙         │
├─────────────────────────────────────────────────┤
│  백엔드 (NestJS + Prisma)                       │
│  → 비즈니스 로직, 인증, 데이터 가공             │
├─────────────────────────────────────────────────┤
│  데이터베이스 (PostgreSQL)                      │
│  → 데이터 저장, 조회, 수정                      │
└─────────────────────────────────────────────────┘
```

## 회고: 프론트엔드 개발자가 전체 구조를 알아야 하는 이유

NEWNDY 프로젝트에서 프론트엔드를 담당하면서, 백엔드의 API 구조를 이해하고 있었기 때문에 효율적으로 협업할 수 있었다.

구체적으로:

- **API 명세 협의** 시 요청/응답 구조를 먼저 제안하여 백엔드 개발자와 빠르게 합의
- **에러 발생 시** 네트워크 탭의 상태 코드와 응답 데이터를 확인해 문제 원인을 빠르게 파악
- **성능 고민** 시 불필요한 API 호출을 식별하고 최적화 방안을 제시

프론트엔드 개발자라고 해서 UI만 다루는 것이 아니라, **전체 시스템의 데이터 흐름을 이해하고 백엔드와 효과적으로 소통하는 능력**이 실무에서 가장 중요한 역량이라고 생각한다.
